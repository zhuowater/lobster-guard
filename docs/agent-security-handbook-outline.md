## 第一篇：安全的物理学根基

### 第 1 章：智能体安全不是传统安全的延伸

**核心论点**: 传统安全是"人写代码，代码有漏洞"。智能体安全是"AI 做决策，决策可被操纵"——攻击面从代码空间移到了语义空间。

- 1.1 **从确定性到概率性**: 传统程序的输入-输出是确定的，LLM 的输出是概率的。安全检查从"精确匹配"变成"统计推断"
- 1.2 **三条数据通道的统一视角**: 入站（IM→Agent）、出站（Agent→IM）、LLM（Agent→模型）——所有安全问题都是这三条管道上的信息流问题
- 1.3 **代理人困境**: Agent 同时是用户的代理人和模型的执行者。当用户指令和模型输出冲突时，谁说了算？这不是技术问题，是治理问题
- 1.4 **龙虾卫士的定位**: 不改 Agent 代码，不改 LLM 模型，只在数据管道上做文章——"网关级安全"

> **实战案例**: 龙虾卫士 v1→v34 的演化——从 40 条正则规则到 12 个理论引擎的认知升级

### 第 2 章：安全退化是物理定律（熵增与耗散结构）

**核心论点**: 安全不是一个状态，是一个过程。就像生命不是物体，是持续对抗熵增的耗散过程。

- 2.1 **热力学第二定律与安全退化**: 封闭系统必然趋向无序——规则老化、攻击变异、配置漂移。你不做任何事，安全就在退化
- 2.2 **Prigogine 耗散结构理论**: 开放系统通过持续输入能量维持远离平衡态的有序。安全系统需要持续的"负熵流"：
  - 红队自进化（v19.0）= 持续注入攻击能量
  - 规则热更新（v3.5）= 持续注入防御能量
  - 密码学审计链（v18.0）= 持续注入信任能量
- 2.3 **安全作为耗散结构的工程实现**:
  - 对抗性自进化闭环（v19.0）: 红队→绕过→新规则→热更新→更强红队——永动机式的安全进化
  - 异常基线检测（v11.2）: 用 7 天滑动窗口建立"正常态"，偏离 2σ 自动告警——Maxwell's Demon
  - 执行信封（v18.0）: 每个决策的数学证明，不可篡改——时间之箭的密码学锚定
- 2.4 **实践推论**: 安全预算不是一次性投资，是持续运营成本。没有持续的能量输入，任何安全架构都会退化为摆设

> **数学框架**: dS/dt > 0（孤立系统）→ 需要 dQ/T < 0（外部负熵流）→ 映射到具体工程措施

### 第 3 章：漏洞是拓扑必然（毛球定理与奇点管理）

**核心论点**: 你不能消灭所有漏洞——这不是能力问题，是数学定理。真正的智慧是选择在哪里留漏洞。

- 3.1 **毛球定理与安全奇点**: 球面上不存在处处非零的连续切向量场——安全系统不可能在所有维度上同时完美。至少有一个"梳不平"的点
- 3.2 **欧拉示性数与奇点预算**: χ(S²) = 2 → 球面上至少有两个奇点（向量场的指标和 = 欧拉示性数）。安全系统的拓扑结构决定了最少需要接受多少弱点
- 3.3 **主动选择奇点**: 与其被动发现漏洞，不如主动选择"在哪里弱"
  - 可配置奇点蜜罐（v18.3）: 在已知弱点处放置带水印的诱饵数据
  - 暴露等级 0-5: 从完全隐身到高价值诱饵
  - 帕累托最优放置: 在（攻击者暴露概率, 真实资产风险）双目标下寻找最优解
- 3.4 **改变拓扑而非对抗拓扑**: 甜甜圈可以梳平（χ(T²)=0），球面不行。当反复遇到同类安全问题时，改变架构（拓扑），不要更努力地"梳毛"
  - CaMeL（v25）就是拓扑变换: 从"检测坏东西"（球面，有奇点）到"只允许好东西"（甜甜圈，无奇点）
- 3.5 **奇点预算仪表盘**: 系统拓扑不变量计算——你至少必须在某处接受弱点，数学告诉你下限是多少

> **代码实现**: `singularity.go`（479行）+ `singularity_budget.go`（182行）——拓扑学在安全工程中的首次系统化落地

### 第 4 章：安全有物理成本下限（Shannon 信息论）

**核心论点**: 检测一个注入攻击需要多少信息？这个问题有物理下限。

- 4.1 **信道容量与检测极限**: Shannon 信道容量定理 C = B log₂(1 + S/N)。攻击检测是一个信号检测问题——攻击信号 vs 正常流量噪声。检测能力有物理上限
- 4.2 **Rate-Distortion 与安全-可用性权衡**: 你不能同时实现零误报和零漏报。就像有损压缩不能同时实现零失真和最小码率。这是信息论定理，不是工程不够好
- 4.3 **三级级联检测的信息论解释**: AC 自动机 → 正则 → 语义模型（v19.1）——粗筛→精筛→深度分析，每一级提升信噪比
- 4.4 **最小描述长度与规则复杂度**: Kolmogorov 复杂度视角——最优规则集是对攻击模式的最短描述。规则太少 = 欠拟合（漏报），规则太多 = 过拟合（误报）
- 4.5 **安全成本不可消灭**: 每一次检测消耗计算资源（延迟 3.1ms）、每一次反事实验证消耗 LLM token（v24）、每一次加密审计消耗存储（v18）。安全是有物理成本的

> **实战数据**: 龙虾卫士生产 P95 延迟 3.3ms，3,040 请求零误报——在 Shannon 极限附近工作

## 第二篇：六大理论支柱

### 第 5 章：因果推断防御（AttriGuard 归因理论）

**核心论点**: 不问"这个输出像不像注入"，而问"这个决策是谁驱动的"——从模式匹配到因果推断的范式革命。

- 5.1 **因果推断 vs 相关性检测**: 传统检测是 P(attack | features)；因果归因是 P(action | do(remove_external_data))——反事实干预
- 5.2 **反事实验证引擎**: 对可疑 tool call，构造"没有外部数据"的对照实验
  - Teacher-Forced Shadow Replay: 锁定之前决策，只改变最新输入
  - Control Attenuation: 梯度衰减外部数据，保留任务信息
  - Fuzzy Survival Criterion: 允许参数级差异（LLM 有随机性）
- 5.3 **归因报告与可解释安全**: 每个被拦截的决策都有完整因果证据链——"为什么拦截"不再是黑箱
- 5.4 **验证成本控制**: 不是每个 tool call 都反事实验证——智能选择策略（路径风险分排序 + 验证缓存 + 预算控制）

> **论文基础**: AttriGuard (arXiv:2603.10749) — 行动级因果归因
> **代码**: `counterfactual.go`（1,358行）+ `cf_adaptive.go` — 静态攻击 0% ASR

### 第 6 章：控制流/数据流分离（CaMeL 执行计划理论）

**核心论点**: LLM 不应该同时理解数据和控制执行——这是操作系统 50 年前就解决的问题（数据/代码分离）。

- 6.1 **根本范式变革**:
  ```
  传统: 用户 → LLM 自己决定做什么 → 执行
  CaMeL: 用户 → 编译成确定性执行计划 → LLM 只填语义槽 → 执行
  龙虾卫士: 用户 → LLM 提出计划 → 龙虾卫士编译+验证 → 执行
  ```
- 6.2 **网关级 CaMeL**: 不改 Agent 代码，在代理层实现——PlanCompiler 将意图编译为 Plan Template
- 6.3 **Capability 权限系统**: 每个数据变量携带权限标签（Sources 并集 + Labels 交集 + TrustScore）
  - 用户输入 → 继承用户 capabilities
  - 外部数据 → 零 capability（不能触发任何操作）
  - LLM 总结 → 输入 capability 的交集
- 6.4 **偏差检测与自动修复**: 实际 tool call 序列 vs 编译的模板——Minor/Moderate/Critical 分级
- 6.5 **与传统 RBAC 的本质区别**: RBAC 是"谁能做什么"，CaMeL 是"这个数据能驱动什么"——权限绑定在数据上，不在身份上

> **论文基础**: CaMeL (Google, arXiv:2503.18813) — AgentDojo benchmark 77% 任务完成 + 可证明安全
> **代码**: `plan_compiler.go`（585行）+ `capability.go`（525行）+ `plan_deviation.go`

### 第 7 章：信息流控制（Fides IFC 理论）

**核心论点**: 数据不仅有内容，还有密级和来源。安全不是检查内容，是控制流向。

- 7.1 **Bell-LaPadula 模型在 AI 时代的复活**: 70 年代军事安全模型的现代化——机密性格 + 完整性格
- 7.2 **双标签系统**:
  ```
  Confidentiality: PUBLIC < INTERNAL < CONFIDENTIAL < SECRET
  Integrity:       TAINT < LOW < MEDIUM < HIGH
  
  传播规则（格运算）:
    合并后机密性取 max（不能泄露）
    合并后完整性取 min（不能被污染驱动）
  ```
- 7.3 **隔离 LLM（Quarantined LLM）**: Fides 独创——被污染数据不直接喂主 Agent，先经隔离 LLM "去污"
- 7.4 **选择性隐藏（Selective Hide）**: 向 LLM 隐藏高机密字段——LLM 看到 `[REDACTED:conf=SECRET]`
- 7.5 **CaMeL vs Fides: ACL vs MLS**: 互补不冲突
  - CaMeL = ACL-like（集合运算）: 数据"从哪来"→ 来源追溯
  - Fides = MLS-like（格运算）: 数据"什么密级"→ 流向控制

> **论文基础**: Fides (Microsoft, arXiv:2505.23643)
> **代码**: `ifc_engine.go`（1,205行）+ `ifc_quarantine.go`（259行）
> **张卓洞察**: LLM Proxy 能看到完整 messages[]，可以在转发前扫描 tool result 做标签驱动隐藏

### 第 8 章：路径感知治理（Runtime Governance 理论）

**核心论点**: 同一个操作，在不同的执行历史下，安全性完全不同。

- 8.1 **从无状态到有状态安全**: 规则不再是 `f(input) → decision`，而是 `f(identity, path, action, state) → violation_prob`
- 8.2 **路径上下文与累积风险**: PathContext 记录完整执行历史——每步贡献增量风险分（权重可配，时间衰减）
- 8.3 **策略类型**:
  - 序列策略: "在 web_fetch 之后 30s 内不允许 send_email"
  - 累积策略: "单次会话 PII 暴露超过 3 个字段 → 升级 block"
  - 降级策略: "路径风险分 > 80 → 后续 tool call 降级为确认模式"
- 8.4 **AI Act 合规策略模板**: 高风险 AI 系统的可解释性要求 + 数据最小化 + 人类监督
- 8.5 **攻防经济学**: 路径感知把攻击成本从 O(1) 提升到 O(n)——攻击者不能只构造一个恶意输入，必须构造完整的执行路径

> **论文基础**: Runtime Governance (arXiv:2603.16586)
> **代码**: `path_policy.go`（920行）

### 第 9 章：密码学信任根（Gödel 不完备与可证明安全）

**核心论点**: 安全系统不能自证安全——这是逻辑定理。但可以用密码学逼近"可证明"。

- 9.1 **Gödel 不完备定理的安全隐喻**: 一致的形式系统无法证明自身的一致性。安全系统无法证明自身的安全——需要外部锚点
- 9.2 **密码学审计链**: 执行信封（HMAC-SHA256 签名）+ Merkle Tree 批量验证——从"我说我记了"到"数学证明我记了"
- 9.3 **不可否认性**: 每个安全决策都有密码学绑定——谁在什么时间对什么输入做了什么决策，不可篡改
- 9.4 **Merkle 批量验证**: 74 批次全过——一次验证覆盖整条审计链
- 9.5 **合规审计的数学基础**: 当监管机构问"你能证明你的 AI 是安全的吗"——执行信封就是答案

> **代码**: `envelope.go` + `merkle.go` — 密码学信任根的工程实现

### 第 10 章：对抗性自进化（进化博弈与 Nash 均衡）

**核心论点**: 安全不是攻防之间的零和博弈，而是动态均衡。最优策略不是"永远赢"，是"让对手的最优策略也是你能接受的"。

- 10.1 **进化博弈论视角**: 攻击者和防御者在策略空间中共同进化。固定策略必然被突破——只有持续进化才能维持均衡
- 10.2 **自进化闭环**: 红队 → 绕过检测 → 提取绕过模式 → 生成新规则 → 热更新 → 更强红队——永动机
- 10.3 **攻击向量变异**: 同义替换 / 编码变形 / 多语言翻译 / 上下文注入——33 个基础向量的无限变体
- 10.4 **贝叶斯自适应**: 基于历史统计预测管理员决策偏好——P(false_positive | user_history)
  - 每个自适应决策生成执行信封，包含完整数学证明（先验+似然+后验+置信区间）
- 10.5 **蜜罐反馈闭环**: 蜜罐收集的真实攻击载荷 → 自动注入自进化引擎作为新种子——攻击者帮你训练防御

> **代码**: `evolution.go`（1,144行）+ `redteam.go` + `adaptive_decision.go`

## 写作风格

- **理论部分**: 数学公式不回避，但每个公式后面跟一句"人话翻译"
- **代码部分**: 只展示核心 20 行，不贴整个文件。用架构图替代实现细节
- **案例部分**: 每个理论章节配一个"如果没有这个机制，攻击者可以这样做"的具体攻击故事
- **语气**: 像 SOUL.md 里说的——"做一个你自己也想跟他说话的人"。不是枯燥的 RFC，是有观点的技术写作
