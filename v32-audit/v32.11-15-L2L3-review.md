## L2: 数据流白盒审查

### v32.11 规则建议队列

**数据生命周期**: evolution.generateAndApplyRules() → SuggestionQueue.Add() → DB INSERT → API List → Accept → RuleEngine.Reload

#### 🔴 P0: Accept 后规则不持久化

`Accept()` 调用 `sq.ruleEngine.Reload(configs, "suggestion-accept")`，这只是内存热更新。
如果进程重启，accepted 的规则会丢失——因为：
1. Accept 把规则 append 到 `GetRuleConfigs()` 的内存快照
2. Reload 更新 AC 自动机
3. 但没有写入 config.yaml 或 conf.d/

**影响**: 审批通过的规则在重启后消失，管理员以为规则生效了但实际没有。
**修复**: Accept 后需要同时 persistRawSection 或写入 conf.d/ 外部 YAML。

#### 🟡 P1: Accept 锁竞争

`Accept()` 持有 `sq.mu.Lock()` 期间调用 `sq.ruleEngine.Reload()`。
如果 Reload 耗时较长（大量规则重建 AC），会阻塞所有 List/Stats/Add 操作。

**影响**: 在高并发演化+审批场景下可能有秒级阻塞。
**修复**: 先 DB 更新，释放锁，再 Reload（Reload 自带自己的锁）。

#### 🟡 P1: List 中 scan error 被静默吞掉

```go
if err != nil {
    continue  // 静默跳过有错误的行
}
```

**影响**: 如果 DB schema 迁移后有不兼容行，用户看到的数量和实际不一致。
**修复**: 至少 log.Printf 记录错误行。

#### 🟢 P2: reviewer 硬编码 "admin"

`api_suggestion.go` 里 `reviewer := "admin"` 写死了。
```go
reviewer := "admin" // TODO: 从 JWT 获取
```

**影响**: 审计日志无法区分谁审批了什么。
**修复**: 从请求 header 或 token 中提取操作者身份。

### v32.13 金丝雀轮换

**数据生命周期**: CanaryRotator.Rotate() → 生成新 token → 更新 cfg 内存 → saveLLMConfig() 写 config.yaml

#### 🔴 P0: Rotate 写 config.yaml 和 LLMProxy 同时使用的竞态

`Rotate()` 修改 `api.cfg.LLMProxy.Security.CanaryToken.Token` 并写磁盘。
但 LLMProxy 检测 Canary 泄露时读的也是 `api.cfg.LLMProxy.Security.CanaryToken.Token`。
如果在 Rotate 过程中（旧 token 已替换、新 token 已写入）有一个 LLM 请求携带旧 token，
会被判断为"未泄露"（因为内存里已经是新 token 了），产生误判。

**影响**: 轮换瞬间有短暂窗口期（< 1ms），旧 canary token 的泄露检测失效。
**修复**: Rotate 后保留旧 token hash 在一个 grace period 列表中（如 5 分钟），检测时同时匹配新旧。

#### 🟡 P1: history 仅在内存，重启丢失

`CanaryRotator.history` 是 `[]CanaryRotationRecord` 内存切片。
重启后 history 为空。

**影响**: 管理员无法追溯历史轮换记录。
**修复**: 写入 SQLite 或者 config.yaml 的 rotation_history 字段。

#### 🟢 P2: saveLLMConfig 的 yaml.Marshal 写法

`persistRawSection` 先读 config.yaml 全文再 unmarshal 为 `map[string]interface{}`，
这会丢失 YAML 注释和格式。

**影响**: 每次 Rotate 都会重写 config.yaml 丢失人工注释。
**修复**: 可接受，但应在文档中说明。

### v32.14 报告定时

#### 🟡 P1: 定时器每分钟触发一次（不是 cron 语义）

`parseInterval()` 硬编码 `return time.Minute`。不管 cron 表达式是什么，都是每分钟触发。

```go
func (s *ReportScheduler) parseInterval() time.Duration {
    return time.Minute
}
```

**影响**: 如果管理员配置 `"0 8 * * *"` 期望每天 8 点，实际每分钟都在生成报告。
**修复**: 要么实现 cron 解析，要么在 start() 里检查当前时间是否匹配 cron 表达式才执行。

#### 🟡 P1: enabled=false 时 start() 不调用，但已在 goroutine 里了

NewReportScheduler 只在 `s.enabled` 时调 `s.start()`，这是对的。
但 `handleReportScheduleUpdate` 修改 `s.enabled = true` 后没有 `s.start()`。

**影响**: 通过 API 启用定时器不生效（必须重启服务）。
**修复**: update 后检查 enabled 状态，按需 start/stop。

### v32.15 引擎开关

#### 🟡 P1: handleEngineToggle 用 json.Decode 读 body，GET 请求没有 body

api.go 里同时注册了 GET 和 POST 到同一个 handler:
```go
case ... && method == "POST": api.handleEngineToggle(w, r)
case ... && method == "GET": api.handleEngineToggle(w, r)
```

GET 请求 body 为空，`json.NewDecoder(r.Body).Decode(&req)` 会失败返回 400。
GET 应该返回当前状态，不是修改状态。

**影响**: GET /engines/plan-compiler/toggle 返回 400 "invalid JSON"。
**修复**: GET 和 POST 分开处理。GET 返回 `{"name":"plan-compiler","enabled":true}`。

## L3: 设计矛盾分析

### 🔴 建议队列 vs 演化引擎的"直接应用"兼容路径

`generateAndApplyRules` 保留了 `else` 分支（没有 suggestionQueue 时直接应用）。
但 main.go 里 **总是** 创建 SuggestionQueue 并赋值给 EvolutionEngine。
所以旧的直接应用路径变成了死代码。

如果有人删了 main.go 的初始化（比如配置项控制 suggestion 开关），
旧路径会复活但没有经过测试验证。

**建议**: 要么删掉旧路径（既然走审批是设计决策），要么加个 config 开关明确控制。

### 🟡 Accept 重载 vs conf.d 规则覆盖

Accept 调用 `Reload(configs, "suggestion-accept")`，其中 configs 来自 `GetRuleConfigs()` 当前内存。
但如果有 conf.d/rules-inbound.yaml 外部文件，按现有逻辑 conf.d 会覆盖代码默认。
Accept 追加的规则在下次 Reload（比如配置变更触发）时会被 conf.d 覆盖掉。

**矛盾**: DB 是 accepted 状态，但规则引擎里没有这条规则。
**建议**: Accept 时同时写入 conf.d/ 文件，确保持久化。

### 🟡 引擎开关 vs 检测链路

`handleEngineToggle` 修改 `cfg.PlanCompiler.Enabled` 等配置字段。
但实际检测链路中是否检查了这些字段？需要确认 proxy.go 里的 planCompiler/capability/deviation 调用是否读 cfg.Enabled 做前置判断。
